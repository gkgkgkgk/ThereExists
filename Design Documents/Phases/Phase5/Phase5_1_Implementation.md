# Phase 5.1 Implementation — Civ-Aware Ship Generation, Persisted
*2026-05-06*

## Purpose
Translate `Phase5_1_Plan.md` into a concrete, commit-by-commit implementation plan detailed enough that a developer (or AI agent) with no prior context can execute Phase 5.1 end-to-end. Do not duplicate design rationale that already lives in the Plan doc — reference it. This doc answers **what code lands in what order and why**.

**Phase 5.1 lands persistence + dial wiring.** No new authored content beyond a single scalar (`ThrustIspBias`) per existing flight archetype. No client work. The civ pipeline itself doesn't change — `GenerateCivilization` already produces the right struct shape; this phase consumes it.

---

## Prerequisites & starting state

**What exists today (tip of `phase-5`):**
- `factory.Civilization` carries `TechProfile { DesignPhilosophy, PreferredCoolingMethods, PreferredIgnitionTypes, PreferredMixtureIDs, AversionToCryogenics, FarDriveFamily, RiskTolerance, ThrustVsIspPreference }`. Generated by `factory.GenerateCivilization`. Not persisted.
- `assembly.GenerateRandomShip(seed int64) (ShipLoadout, error)` — civ-blind. Picks Short / Medium / Far slots via `flight.GenerateForSlot`, which gates on `GenericCivilization`'s TechTier (=3) for every roll.
- `flight.archetypeEntry` already has `rarity float64` for weighted pick and `minTechTier int` for tier gating (Phase 4 commit 6 / commit 8). The civ-bias multipliers attach to these existing weights — no new selection mechanism.
- `db.Migrate` runs idempotent `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` in `server/internal/db/db.go`. New migration statements append to the same slice.
- `POST /api/ships/generate?player_id=X[&seed=N]` (`handlers/ship.go`) looks up the player's active ship row, reads or overrides the seed, calls `assembly.GenerateRandomShip`, persists the loadout JSON onto the ship row.
- `GET /api/player` (`handlers/player.go`) provisions a player + an empty `ships` row. No civ.
- `FactoryVersion = "phase5-v1"` in `assembly/ship.go`.

**Phase 5.1 lands:**
1. **Migration:** `civilizations` table; nullable `civ_id` on `players`; nullable `civ_id` on `ships`.
2. **`factory.SaveCivilization` / `LoadCivilization`** — JSONB round-trip for `TechProfile` and `Planet`.
3. **`GenerateRandomShip(seed, civ *Civilization)` signature change.** `nil` falls back to current behavior.
4. **`ThrustIspBias float64` field on each flight archetype** with hand-authored values per Plan §5.
5. **Civ-aware archetype weighting** — `ThrustVsIspPreference`, `RiskTolerance` (rarity sharpening), `PreferredCoolingMethods` / `PreferredIgnitionTypes` soft bias.
6. **Civ-aware mixture pick** — `PreferredMixtureIDs` × 3.0 boost, `AversionToCryogenics` damping on cryogenic mixtures.
7. **`HealthInitRange` shifting via `RiskTolerance`** in the per-archetype roll.
8. **Wire civ generation + persistence into `POST /api/ships/generate`.** First call generates and persists a civ; subsequent calls load it.
9. **`FactoryVersion = "phase5_1-v1"`** + civ-divergence sweep test.

**Out-of-scope** (Plan §6): manufacturer-from-civ, cross-civ ship capture, civ pregeneration, civ regeneration endpoint, client work, `DesignPhilosophy` reading by ship gen.

## Commit-by-commit plan

### Commit 1 — `db: civilizations table + civ_id FKs`
**Intent.** Land the schema. No code reads or writes the new columns yet — the rest of Phase 5.1 builds on this.

**Files modified.**
- `server/internal/db/db.go` — append to the `stmts` slice in `Migrate`:
  ```sql
  CREATE TABLE IF NOT EXISTS civilizations (
      id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      name            TEXT NOT NULL,
      description     TEXT NOT NULL,
      homeworld_desc  TEXT NOT NULL,
      age_years       BIGINT NOT NULL,
      tech_tier       INTEGER NOT NULL,
      flavor          TEXT NOT NULL,
      profile         JSONB NOT NULL,
      planet          JSONB NOT NULL,
      factory_version TEXT NOT NULL,
      created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );
  ALTER TABLE players
      ADD COLUMN IF NOT EXISTS civ_id UUID
          REFERENCES civilizations(id) ON DELETE SET NULL;
  ALTER TABLE ships
      ADD COLUMN IF NOT EXISTS civ_id UUID
          REFERENCES civilizations(id) ON DELETE SET NULL;
  ```

**Gotchas.**
- `ON DELETE SET NULL` (not CASCADE). A civ being deleted should not vaporise the player or their ship; their `civ_id` just becomes NULL and the next ship roll re-provisions a civ.
- Nullable columns. Existing rows that pre-date Phase 5.1 keep `civ_id = NULL`. The next ship-roll attempt for those players triggers fresh civ provisioning (commit 8).
- No data migration / backfill. Lazy provisioning is intentional and cheaper.
- Restart the server against an existing DB to confirm `Migrate` runs idempotently — second-run no-op.

### Commit 2 — `factory: civstore Save/Load with JSONB round-trip`
**Intent.** Persistence helpers. Lives in `factory/` next to the type it persists, takes a `*sql.DB` so it doesn't depend on a package-global connection. No HTTP layer, no imports into `handlers/`.

**Files created.**
- `server/internal/factory/civstore.go`:
  - `func SaveCivilization(ctx context.Context, db *sql.DB, civ *Civilization, planet *Planet, factoryVersion string) error`:
    - Marshal `civ.TechProfile` to JSON.
    - Marshal `planet` to JSON.
    - `INSERT INTO civilizations (id, name, description, homeworld_desc, age_years, tech_tier, flavor, profile, planet, factory_version) VALUES ($1, ..., $10)`.
    - Use `civ.ID` as the primary key (already a `uuid.NewString()` from `GenerateCivilization`). On conflict, return an error — caller decides whether to retry with a new ID. Don't ON CONFLICT DO NOTHING; that hides bugs.
  - `func LoadCivilization(ctx context.Context, db *sql.DB, id string) (*Civilization, *Planet, error)`:
    - `SELECT id, name, description, homeworld_desc, age_years, tech_tier, flavor, profile, planet FROM civilizations WHERE id = $1`.
    - On `sql.ErrNoRows`, return a sentinel `ErrCivNotFound` so callers can distinguish missing-civ from infrastructure errors.
    - Unmarshal JSON columns into `TechProfile` and `Planet`.
    - Build and return the `Civilization` (with `HomeworldPlanetID = nil` — still reserved per Phase 5).

**Files created (tests).**
- `server/internal/factory/civstore_test.go`:
  - Skip if `TEST_DATABASE_URL` env var is unset (so CI without a DB still passes; local dev runs against the dev Postgres).
  - `TestSaveLoadRoundTrip`:
    - Use `sql.Open` against `TEST_DATABASE_URL`, run `db.Migrate`.
    - Build a `*Civilization` with every `TechProfile` field populated (including `RiskTolerance`, `ThrustVsIspPreference`, non-empty preference arrays).
    - Build a `*Planet` via `GeneratePlanet(42)`.
    - Save, then Load.
    - Assert deep equality of every field on both struct types. Use `reflect.DeepEqual` or a hand-crafted comparator.
  - `TestLoadCivNotFound`: assert `errors.Is(err, ErrCivNotFound)` on a fresh UUID.

**Gotchas.**
- The JSONB columns need explicit `[]byte` marshaling. Use `json.Marshal` to bytes, pass as `string($1::jsonb)` or `[]byte` directly — pgx handles both.
- When unmarshaling `Planet`, you'll lose the in-memory enum pointer indirection (the JSON tags on `Planet` already produce a valid round-trip — see `planet.go`). Verify with the round-trip test.
- Don't introduce a `Civilizations` cache in this commit. Phase 5.1 stays load-on-demand; caching is its own concern.
- `civstore.go` lives in `package factory` — no new subpackage. The function names start with `Save` / `Load` so they read clearly at call sites.

### Commit 3 — `assembly: thread *Civilization through GenerateRandomShip`
**Intent.** Add the parameter, plumb it nil-safe into `flight.GenerateForSlot`. No behavior change for existing callers. Tests pass with `nil`.

**Files modified.**
- `server/internal/factory/assembly/ship.go`:
  - Change signature: `func GenerateRandomShip(seed int64, civ *factory.Civilization) (ShipLoadout, error)`.
  - When `civ != nil`, use `civ.ID` and `civ.TechProfile` for downstream calls. When `nil`, use `GenericCivilizationID` and treat the TechProfile as zero-valued (the existing behavior).
- `server/internal/factory/flight/flight.go`:
  - Extend `GenerateForSlot` signature to accept a `*factory.Civilization`. **But** — `flight` cannot import `factory` (import cycle; that's why `ManufacturerPicker` and `CivTechTierLookup` are injected). So introduce a tiny opaque type the dispatcher can read without importing the civ struct directly:
    ```go
    // CivBias is the subset of Civilization fields the flight dispatcher
    // needs. Defined in flight/ to keep the package leaf-y. The factory
    // root constructs and passes it.
    type CivBias struct {
        TechTier                int
        RiskTolerance           float64
        ThrustVsIspPreference   float64
        AversionToCryogenics    float64
        PreferredMixtureIDs     map[string]bool   // set, not slice — picker does O(1) lookups
        PreferredCoolingMethods map[string]bool   // CoolingMethod.String() keys
        PreferredIgnitionTypes  map[string]bool   // IgnitionMethod.String() keys
    }
    ```
  - New signature: `func GenerateForSlot(slot FlightSlot, civilizationID, previousManufacturerID string, civ *CivBias, rng *rand.Rand) (FlightSystem, string, error)`. `civ == nil` keeps the current uniform-bias behavior; the existing tier gate falls back to `civTechTierLookup(civilizationID)` so the no-civ path still works.
- `server/internal/factory/assembly/ship.go`:
  - Add a private `civBiasFor(civ *factory.Civilization) *flight.CivBias` helper that builds the `CivBias` from `civ.TechProfile` (including the enum.String() conversions for the cooling/ignition sets). Returns `nil` for `nil` civ.
  - Pass through to each `flight.GenerateForSlot` call.

**Files modified (tests).**
- `assembly/ship_test.go`, `assembly/ship_sweep_test.go`: update existing call sites to pass `nil`. Sweep tests should still pass — they're proving the no-civ path works.
- `handlers/ship.go`: existing call site updates to pass `nil` civ for now (commit 8 wires the real civ).

**Gotchas.**
- Do NOT import `factory` from `flight`. The `CivBias` indirection is the price of avoiding the import cycle. The factory root knows about both.
- Mutex-free: `CivBias` is read-only after construction. Each ship-roll gets its own `CivBias` (built in `assembly`) and never mutates it.
- The `CivBias.PreferredMixtureIDs` is a `map[string]bool` (set semantics) so the per-mixture pick can do O(1) `if civ.PreferredMixtureIDs[id]` checks.
- The `CivBias.PreferredCoolingMethods` keys are the `CoolingMethod.String()` output (e.g. `"regenerative"`), not the int enum value. Same for ignition. This keeps the `flight` package free of enum imports.

### Commit 5 — `factory/flight: civ-aware archetype weighting`
**Intent.** When `*CivBias` is non-nil, multiply the existing rarity weight by:
1. `ThrustVsIspPreference` proximity (per Plan §5);
2. `RiskTolerance` rarity-sharpening (low risk → power-up the weights toward common; high risk → flatten);
3. `PreferredCoolingMethods` overlap (× 1.5);

…before the weighted sample. Tier gating stays exactly as it is.

**Files modified.**
- `server/internal/factory/flight/flight.go`:
  - In `GenerateForSlot`, after the tier-filtered `eligible` list is built, add a new pass that computes per-entry weight:
    ```go
    weight := e.rarity
    if weight <= 0 { weight = 1.0 }
    if civ != nil {
        // ThrustVsIspPreference: closer = boost.
        diff := math.Abs(civ.ThrustVsIspPreference - e.thrustIspBias)
        weight *= 1.0 + 0.8*(1.0 - diff/2.0)

        // RiskTolerance: power-up rarity weights for low tolerance.
        // weight^(1 + (1 - risk)) — preserves rank order, sharpens
        // the distribution toward higher-rarity (common) archetypes
        // when risk is low; flattens when risk is high.
        // Use weight^risk for the high-risk side via a single expr:
        //   exp = 1 + (1 - 2*risk)
        // risk=0.0 → exp=2 (sharp); risk=0.5 → exp=1 (no-op);
        // risk=1.0 → exp=0 (flat — every option equal).
        // Clamp exp to [0.1, 3] to avoid degenerate distributions.
        exp := math.Max(0.1, math.Min(3.0, 1.0 + (1.0 - 2.0*civ.RiskTolerance)))
        weight = math.Pow(weight, exp)

        // PreferredCoolingMethods: 1.5× when archetype's allowed
        // cooling methods overlap the civ's preferences. Requires
        // the archetype generator to expose its allowed cooling
        // methods to the dispatcher — see helper below.
        if archetypeFavoursCooling(e, civ.PreferredCoolingMethods) {
            weight *= 1.5
        }
    }
    if weight < 0.01 { weight = 0.01 }   // floor — never zero out
    ```
  - Add an interface (or accessor closure) `func (archetypeEntry) AllowedCoolingMethodNames() []string` so the dispatcher can ask without importing flight subtypes. Each archetype struct (`LiquidChemicalArchetype`, `RelativisticDriveArchetype`) implements it by calling `.String()` on its `AllowedCoolingMethods` enum slice. Wire it into `archetypeEntry` at registration time as a `coolingNames []string` field, populated by the registering category package. Avoid runtime reflection.
- `server/internal/factory/flight/liquid.go`, `far.go`: at registration, pre-compute the cooling-method name slice for each archetype and pass it into `registerFull`.

**Files modified (tests).**
- `flight/liquid_test.go` (or new `flight/civbias_test.go`): drive `GenerateForSlot` with two synthetic `CivBias` (low-risk vs high-risk; thrust-leaning vs Isp-leaning) over 1000 seeds each. Assert the distribution shifts in the expected direction. Use a Chi-squared-ish "did rare archetype X appear more often under high risk?" check, not bit-equality.

**Gotchas.**
- All math is multiplicative. Order of multipliers doesn't matter, but the floor at `0.01` does — without it, an extreme civ profile could zero out an archetype entirely and `len(eligible) > 0` doesn't guarantee a non-zero total weight.
- The `RiskTolerance` formula above is one possible shape. The Plan §5 description ("sharpens or flattens") is satisfied by any monotonic function of risk; this `1 + (1 - 2*risk)` exponent is the simplest one with the right endpoints. Document the formula in a comment so future-me doesn't second-guess.
- `PreferredIgnitionTypes` is **not** wired here — it's a property of the rolled mixture, not the archetype. Lands in commit 6.

### Commit 6 — `factory/flight: civ-aware mixture pick`
**Intent.** Replace the uniform mixture pick (currently `rng.Intn(len(allowed))`) with a weighted pick that consults `PreferredMixtureIDs`, `AversionToCryogenics`, and `PreferredIgnitionTypes`.

**Files modified.**
- `server/internal/factory/flight/liquid.go`, `far.go`:
  - Locate the mixture-pick line in each generator (it's a `rng.Intn` over `allowedMixtureIDs`).
  - Replace with a helper:
    ```go
    func pickMixture(allowed []string, civ *CivBias, rng *rand.Rand) string
    ```
    - Build per-mixture weight starting at 1.0.
    - `if civ != nil`:
      - If `civ.PreferredMixtureIDs[id]`, weight × 3.0.
      - Look up the mixture (`factory.LookupMixture(id)` — but flight can't import factory; see Gotchas) — actually use a shim function injected from the factory root, mirroring `manufacturerPicker`. Call it `MixtureLookup func(id string) (cryogenic bool, ignitionMethod string, ok bool)`.
      - If cryogenic, weight × `(1.0 - civ.AversionToCryogenics)`.
      - If `civ.PreferredIgnitionTypes[ignitionMethod]`, weight × 1.5.
    - Floor at 0.05 (mixture floor is higher than archetype floor — fewer options per slot, rounding to zero is more visible).
    - Weighted sample.
- `server/internal/factory/flight/flight.go`:
  - New injection: `type MixtureBiasLookup func(mixtureID string) (cryogenic bool, ignitionMethod string, ok bool)`. Add `SetMixtureBiasLookup`.
- `server/internal/factory/factory.go` or wherever startup wiring happens (`cmd/server/main.go`):
  - Wire `flight.SetMixtureBiasLookup` to a closure that calls `factory.LookupMixture` and returns `m.Cryogenic, m.Ignition.Resource.... .String()` — actually `IgnitionMethod` lives on the mixture's `Config`, not the resource. Confirm where the per-mixture ignition method comes from and surface a stable string. If the mixture itself doesn't carry an `IgnitionMethod` directly, derive it (`Hypergolic == true` → `"hypergolic"`; otherwise read from `Ignition.Resource.Category`, etc.). Document the mapping in a comment.

**Gotchas.**
- The flight package's existing tests don't construct mixtures from outside; they trust the registry. The new `MixtureBiasLookup` wiring needs a default for tests (a no-op closure that returns `false, "", false` — caller treats `!ok` as "no bias info, weight stays at 1.0").
- `PreferredIgnitionTypes` was *meant* to bias archetypes (per Plan §5: "soft archetype-selection bias"), but it's actually a property of the mixture, not the archetype — same archetype with two different mixtures has two different ignition methods. The Plan was imprecise. Wire it on the mixture pick (here), not the archetype pick (commit 5). Update Plan §5 to match if it's confusing.
- Don't try to combine the cooling and ignition biases into one path. They live at different levels (archetype vs mixture) for a reason.

### Commit 7 — `factory/flight: HealthInitRange shifting via RiskTolerance`
**Intent.** When the archetype rolls `Health`, the lower bound shifts based on civ risk tolerance. Low-risk civs always start near top; high-risk civs accept the archetype's full range.

**Files modified.**
- `server/internal/factory/flight/liquid.go`, `far.go`:
  - In each archetype's `Generate*` function, locate the `HealthInitRange` roll. It's currently `rng.Float64() * (hi - lo) + lo`.
  - Replace with `rng.Float64() * (hi - effectiveLo) + effectiveLo`, where:
    ```go
    effectiveLo := lo
    if civ != nil {
        effectiveLo = lo + (hi-lo)*(1.0-civ.RiskTolerance)
    }
    ```
    - `risk = 0.0` → `effectiveLo = hi` (always rolls top of range).
    - `risk = 1.0` → `effectiveLo = lo` (full range, original behavior).
    - `risk = 0.5` → midpoint of range.
- The archetype generators already receive `(manufacturerID, rng)` per `archetypeGenerator` typedef. They need access to `*CivBias` too. Extend the typedef:
  ```go
  type archetypeGenerator func(manufacturerID string, civ *CivBias, rng *rand.Rand) (FlightSystem, error)
  ```
  - All registered generators update to accept (and mostly ignore) the parameter; only the health-roll lines actually read it.
  - `GenerateForSlot` passes `civ` through after the archetype pick.

**Gotchas.**
- The signature change is wide (every archetype generator). Necessary — without `civ` reaching the per-archetype roll, neither this commit nor future per-archetype dial readouts can land.
- For `RBCABeamCore` (Far drive, intentionally fragile per Plan §1: "spawns damaged — the drive *is* the adventure"), `RiskTolerance` shifting matters most. A risk-averse civ rolling RBCA still rolls a fragile drive — the floor of `[0.40, 0.60]` is intentionally low — but they get the high end of that low range.

### Commit 8 — `handlers/ship: civ provisioning + persistence on POST /api/ships/generate`
**Intent.** First call for a civ-less player generates a civ, persists it, links it to the player, then rolls a civ-aware ship. Subsequent calls load the existing civ.

**Files modified.**
- `server/internal/handlers/ship.go`:
  - Add an `llm.Client` field to `ShipHandler` (constructed from `NewOpenAIClient` at startup; nil-safe — civ provisioning 503s if the client is nil and `civ_id` is null).
  - In `Generate`:
    1. Look up the player → `seed`, `active_ship_id`, `civ_id sql.NullString`. (Existing query gets a new column.)
    2. If `civ_id.Valid == false`:
       - Return 503 if `h.llm == nil` (mirrors the civ endpoint behavior).
       - `civ, planet, err := factory.GenerateCivilization(ctx, h.llm, seed)`.
       - `factory.SaveCivilization(ctx, h.db, civ, planet, assembly.FactoryVersion)`.
       - `UPDATE players SET civ_id = $1 WHERE id = $2`.
       - Continue with this fresh civ.
    3. Else:
       - `civ, _, err := factory.LoadCivilization(ctx, h.db, civ_id.String)`.
    4. Resolve seed (existing logic).
    5. `loadout, err := assembly.GenerateRandomShip(seed, civ)`.
    6. Persist loadout AND `ships.civ_id = civ.ID`. Existing `UPDATE ships SET loadout = $1, factory_version = $2 WHERE id = $3` becomes `UPDATE ships SET loadout = $1, factory_version = $2, civ_id = $3 WHERE id = $4`.
  - Wrap steps 1–3 (and the player UPDATE) in a transaction to avoid orphan civs if the link fails.
- `server/internal/handlers/ship.go` (constructor):
  - `NewShipHandler(db *sql.DB, llmClient llm.Client) *ShipHandler`.
- `server/cmd/server/main.go`:
  - Pass `llmClient` to `NewShipHandler` (already constructed for `NewCivilizationHandler`).

**Files modified (tests).**
- `handlers/ship_test.go`: existing tests use a mock player + mock ship row. Update them to either (a) pre-seed `civ_id` on the player and a matching civilizations row, or (b) inject a fake `llm.Client` so the test exercises the full provisioning path. Recommend (b) for one new test (`TestShipHandler_GeneratesCivOnFirstCall`) and (a) for the existing happy path.
- New: `TestShipHandler_LoadsExistingCivOnSubsequentCall` — pre-seed civ + player.civ_id, assert no LLM calls.

**Gotchas.**
- Transaction semantics. If `SaveCivilization` succeeds but the `players.civ_id` UPDATE fails, you get an orphan civ. Either (a) wrap both in `BeginTx`, or (b) accept orphans (they're rare, no FK references them, manual cleanup is fine). Recommend (a) for cleanliness.
- The `seed` resolution path defaults to the player's seed. Civ generation uses that same seed for `GeneratePlanet` — so the same player always generates the same planet (and roughly the same age) on retry, but the LLM steps remain non-deterministic. Acceptable.
- Don't regenerate the civ on every `POST /api/ships/generate` call. Only when `civ_id` is null. This is the deliberate "civ is permanent" decision from Plan §6.
- Watch for the ship handler's existing `?seed=` override path. The seed used for civ generation should be the player's persistent seed, not the override — overriding the seed for ship rolls is for "let me see different loadouts for this same civ/player," and changing the civ would defeat that.

### Commit 9 — `assembly: FactoryVersion phase5_1-v1 + civ-divergence sweep test`
**Intent.** Tag the version. Add a tripwire test that proves civ-aware generation actually produces visibly different ships across civs.

**Files modified.**
- `server/internal/factory/assembly/ship.go` — `const FactoryVersion = "phase5_1-v1"`.

**Files created.**
- `server/internal/factory/assembly/ship_civ_sweep_test.go`:
  - Build two synthetic civilizations programmatically (no LLM):
    - **Conservative civ:** `RiskTolerance: 0.05`, `ThrustVsIspPreference: -0.5`, `PreferredMixtureIDs: ["LOX_LH2"]` (or whatever balanced choice exists), `AversionToCryogenics: 0.0`, `TechTier: 3`.
    - **Wild civ:** `RiskTolerance: 0.95`, `ThrustVsIspPreference: +0.7`, `PreferredMixtureIDs: ["MMH_NTO"]`, `AversionToCryogenics: 0.8`, `TechTier: 3`.
  - For each civ, generate 200 ships across seeds [1..200].
  - Assert:
    - Wild civ's RDE / SABRE / PDD archetype frequency is at least 2× the conservative civ's.
    - Conservative civ's mean ship Health (across all flight slots) is higher than wild civ's by a meaningful margin.
    - Conservative civ's mixture pick is dominated by its preferred ID; wild civ's spreads more.
  - Also: `TestNoCivBackwardCompat` — `GenerateRandomShip(seed, nil)` over 200 seeds still passes the existing sweep invariants from Phase 4 (Short and Medium populated, Far gated by `GenericCivilization` tier).
  - `TestFactoryVersionPhase5_1` — trivial.

**Gotchas.**
- Synthetic civs in this test are built directly as `*Civilization` literals — no DB, no LLM. Keeps the test fast and focused on the dial-wiring layer.
- The "2× more often" assertion is loose on purpose. Sampling noise over 200 seeds means strict counts will flake; structure the assertion as `wildRareCount > conservativeRareCount * 2 - tolerance`.
- If the wild/conservative divergence isn't visible, the dial multipliers in commits 5–7 need retuning. Plan §10 flags this; the divergence test is the canary.

## Post-phase verification (manual)

1. Fresh DB. `go test ./server/...` — all green.
2. `swag init -g cmd/server/main.go -o api` to refresh the spec (no new endpoints, just smaller things possibly drifted).
3. Start server with `OPENAI_KEY` set.
4. `GET /api/player` → returns a fresh player with a ship row, `civ_id` null.
5. `POST /api/ships/generate?player_id=<id>` (first call):
   - Logs show one `GenerateCivilization` call (3 LLM round trips).
   - Response: ship loadout JSON, `factory_version: phase5_1-v1`.
   - DB: `civilizations` has one new row; `players.civ_id` set; `ships.civ_id` matches.
6. `POST /api/ships/generate?player_id=<id>` (second call, same player, no `?seed=`):
   - No LLM calls (logs).

## Deferred-but-spec'd items (extension points)

- **Manufacturer-from-civ.** Land when civs spawn manufacturer rosters; until then, manufacturer pool is shared.
- **Cross-civ ship capture / transfer.** `ships.civ_id` is the hook; no mechanic reads it differently from `players.civ_id` yet.
- **Civ pregeneration / catalog.** Batch job that pre-rolls 100+ civs, indexed for player onboarding.
- **Civ regeneration endpoint.** `POST /api/players/:id/regenerate-civ` for testing prompt changes against existing players.
- **`DesignPhilosophy` reading at ship-gen time.** Currently consumed only by civgen prompts; could later bias archetype rarity via keyword matching. Document as fragile.
- **Caching `LoadCivilization`.** Today every ship-roll hits the DB. If that becomes hot, add an in-memory LRU.
- **Migration of pre-Phase-5.1 ships.** None today. If the project ever ships to staging, write a one-shot migration that backfills `ships.civ_id` from `players.civ_id` for non-null player civs.
